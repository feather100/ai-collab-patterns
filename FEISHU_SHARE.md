# AI 协作规范 — 飞书分享卡片

---

## 📌 一句话

**多 AI 协同最大的风险是不一致与不可控。** 这套协议用三个文件解决：AI 入口（新 AI 知道从哪读起）+ 决策记录（不重推"为什么"）+ 协作规范（SoT、成本闸、验收标准）。

## 📂 仓库地址

> https://github.com/feather100/ai-collab-patterns

## 📦 三个核心文件

| 文件 | 解决什么问题 |
|---|---|
| `AI_ENTRY.md` | AI 版的 README — 新 AI 进来先读这个，知道 SoT 在哪、约束在哪、怎么汇报 |
| `DECISIONS.md` | CHANGELOG 的决策版 — 每个关键决策记下"为什么"，后来者不用重推 |
| `COLLAB.md` | AI 版的 eslint — 变更流程、验收标准、成本确认、文档巡检 |

## 🧠 三条铁律

1. **📝 文档留痕** — 任何设计决策都要记录理由
2. **💰 成本确认** — 任何付费/外部 API 动作必须先停下说明开销，绝不默认执行
3. **🎯 SoT 一致** — 改标准 ⟺ 更新决策记录 ⟺ 更新下游，三者不同步视为未完成

## 🚀 快速上手（5 分钟）

1. 把三个模板文件复制到项目根目录
2. 替换 【占位符】（项目名、技术栈、目录结构）
3. 在 README.md 顶部加一行：`> 任何 AI 在改动本仓库前，必须先读 AI_ENTRY.md`
4. 由负责人写前 3 条决策记录作为"气候设定"

## 💡 来自实战

这套模式来自 IAIBench 项目（工业 AI Agent 可信评测平台），已在该项目运行多个月，经过 Claude / Codex / DeepSeek / GPT 等多个 AI 的验证。

已验证有效的效果：
- ✅ 新 AI 接入后第一次改的东西在预期范围内
- ✅ AI 会在改标准前先写决策记录
- ✅ 决策记录被后来者引用（"如 DR-003 所述"）
- ✅ AI 在调用外部 API 前会停下询问成本
