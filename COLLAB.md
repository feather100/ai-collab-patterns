---

## 1. 为什么需要这份规范

**多 AI 协同最大的风险是不一致与不可追溯**——A 改了标准、B 不知道，C 绕过事实源直接改了下游。本规范确保：无论谁进来，都从同一个事实源出发、按同一套流程留痕、按同一套标准验收。

---

## 2. 唯一事实源（SoT）与一致性铁律

| 层 | 文件 | 谁说了算 |
|---|---|---|
| **标准（机器可读）** | `【路径】` | **核心约束以此为准** |
| **决策记录（人读）** | `【路径】` | 每条决策为什么这么定 |
| **协作规范** | `【路径】`（本文件） | 流程与验收 |
| **【其他层】** | `【路径】` | 【说明】 |

**一致性铁律：** 改 SoT ⟺ 必在决策记录追加/更新 ⟺ 必让下游自检通过。三者任一不同步，本次改动视为**未完成**。

---

## 3. 入会检查（任何 AI 开工前）

1. 读 `AI_ENTRY.md` + 本文件。
2. 读决策记录，理解现状由来。
3. 判定本次改动属于哪一层，**先改 SoT，再改下游**。
4. 若涉及标准变更，先在决策记录写清理由，再动 SoT。

---

## 4. 变更流程

```
读 SoT + 决策记录
   → 确定改动层级
   → （若改标准）先写决策记录 → 改 SoT → 改下游实现
   → 跑验收（见 §6）
   → 在 docs/ 留痕（决策 + 过程 + 暴露的问题）
   → 提交（见 §7）
```

**留痕要求：** 不只记结论，也记**过程与暴露的问题**。

---

## 5. 角色分工（建议，非强制）

多 AI 协同时按擅长领域分工，避免互相覆盖：

| 工种 | 典型职责 | 主要触及 |
|---|---|---|
| 【工种1】 | 【职责】 | 【文件/目录】 |
| 【工种2】 | 【职责】 | 【文件/目录】 |
| ... | ... | ... |

**协同礼仪：** 跨层改动前，在对应文档留一条意图说明；不静默覆盖他人产出；发现 SoT 与下游不一致，先记录再修。

---

## 6. 验收标准（Definition of Done）

收工前**逐条自检**，任一不过即未完成：

- [ ] 【核心自检】通过（**零外部调用、零费用**）
- [ ] 若动了【某层】：【专项自检】通过
- [ ] 若动了标准/规则：SoT 与决策记录已同步
- [ ] 本次决策、过程、暴露的问题已在 `docs/` 留痕
- [ ] **未引入未经人类许可的付费 / 外部 API 调用**
- [ ] 自述本次改了哪些 SoT 层，下游是否已同步

---

## 7. 提交与版本规范（git）

- 提交信息讲清**改了什么 + 为什么**，关联决策记录编号（如 `core: 重构路由 (DR-003)`）
- 不提交 `node_modules/`、`dist/`、本地密钥
- **绝不提交任何 API key 或密钥**到仓库
- 标准类改动与代码改动尽量分提交，便于审阅与回滚

---

## 8. 会话交接（多机器 / 多 AI 接力）

### 8.1 为什么需要

Reasonix（或其他 AI 协作工具）的会话历史和记忆存在本地机器，**换机器后不会自动同步**。代码走 GitHub 可以跨机器，但 AI 的对话上下文会断。`SESSION_HANDOFF.md` 是解决这个问题的轻量方案。

### 8.2 怎么做

**每次收工前**（或切换机器前），AI 执行：

```
1. 写 SESSION_HANDOFF.md（模板见仓库根目录）
2. git add SESSION_HANDOFF.md
3. git commit -m "handoff: SESSION-xxx 完成内容摘要"
4. git push
```

**每次开工后**（在新机器上），AI 执行：

```
1. git pull
2. 读 SESSION_HANDOFF.md — 理解当前状态
3. 读 DECISIONS.md — 理解关键决策
4. 读 AI_ENTRY.md — 确认 SoT 和约束
5. 开始干活
```

### 8.3 会话交接模板内容

| 字段 | 用途 |
|---|---|
| 本次完成内容 | 上一会话做了什么 |
| 当前状态快照 | 分支、commit、自检结果 |
| 关键上下文 | 下一会话必须知道的事、踩过的坑 |
| 待办/下一步 | 优先级排序的后续任务 |
| 相关决策记录 | 本次新增或更新的 DR 编号 |
| 成本记录 | 本次是否产生费用 |

### 8.4 原则

- **每次写交接不超过 5 分钟** — 不是写文档，是缓存上下文
- **交接文件提交到 Git** — 走 GitHub 跨机器，零额外成本
- **交接内容比交接格式重要** — 关键是"下一会话必须知道什么"，不是格式完美

---

## 9. 硬性约束：成本动作先确认

任何**产生费用 / 调外部 API / 拉大体量内容**的动作，执行前必须先停下，讲清开销（调用次数、模型、预估 token/费用、影响范围），**得到人类明确许可后再做，绝不默认执行**。

- 真花钱的代码路径必须带**硬性成本闸**
- 默认走**零成本路径**
- 详见决策记录 §0.1

---

## 10. 文档巡检（Document Inspection）

所有核心文档必须定期或在重大架构变更后进行巡检，确保历史设计与最新版本无矛盾。巡检后需在文档头部标注 `> 最后巡检日期: YYYY-MM-DD`。
